# CodeKids 企业级平台（MVP 简明说明）

概述

MVP 范围声明（两个月内上线）：本阶段聚焦“登录/注册、微信支付、编辑器与可视化、课程上传（三种模式，隐藏课程代码（类似codemonkey学生使用写好的函数进行编程），隐藏上下代码，学生补全中间，自由创作）与学习（含视频，ppt，图片）、AI 助手（报错解释/翻译/示例建议）”。PK、天气/生态、成就商城、家长观战等延后到后续版本渐进发布。



总体技术栈（MVP 推荐）

八、平台技术架构（中国大陆 AI 接入 · MVP 版）



1️⃣ 前端技术栈（单通道优先 · 协作灰度） 

React + TypeScript + Vite + Tailwind CSS + Zustand + Framer Motion + Lottie + Monaco Editor + 原生 WebSocket（单业务通道）+ 自定义 Yjs Provider（同路复用）/ y-websocket（灰度） + ReactPlayer + PDF.js + 微信小程序 / H5



仅一条业务 WS（除 HMR 外）。协作默认同路复用：自定义 Yjs Provider 将协作更新封装为 topic=collab.\* 的消息；

灰度开关：必要时启用第二条 y-websocket 专线（仅协作），不影响“单业务 WS”SLO统计。



可视化窗口：固定宽高比、无拉伸；前端按固定时间步长渲染，支持“帧丢弃优先于排队”的策略，避免积压卡顿。



输入增强：中文符号检测与一键修复；报错翻译/朗读；Lint 与英语术语提示。



版本回退：保存点生成“缩略图+说明”，与后端快照对齐（见 §3）。



长回复流式：AI 输出统一分片协议，前端按序拼接；编辑器不阻塞。



2️⃣ 后端技术栈（统一栈+清晰职责）



Java（Spring Boot + WebFlux / Reactor Netty 网关） + FastAPI（HTTP 微服务） + Celery（任务队列） + Redis + RabbitMQ + Docker + Kubernetes



对外统一网关：WebFlux（Reactor Netty）承载原生 WS、鉴权、粘性会话、一致性哈希分发、背压与限流。



微服务统一：HTTP 全用 FastAPI；如需 Socket.IO 生态，仅作为后端内部侧边车，不对外暴露。



可靠广播：业务事件（公告/房间消息/PK状态）用 RabbitMQ topic/fanout；

Presence/路由用 Redis（短状态+TTL），不以 Redis Pub/Sub 充当可靠总线。



AI Router：统一接通义/文心/星火（MVP 先接一家），内建配额/QPS、熔断、降级与审计。



3️⃣ 可视化执行链路（编辑器核心·Pygame 专段）



服务组件：py-exec（FastAPI） + 隔离沙箱（Docker/namespace + seccomp + 资源限额） + Celery Worker

渲染方式：无头渲染（SDL\_DUMMY/OSMesa）→ 帧采集（Surfaces）→ 编码 → WS 推送



执行安全：每次运行在独立容器/进程内；默认禁网、限制 CPU/内存/文件系统；白名单模块。



帧编码：



MVP：关键帧 PNG + 差分帧（RLE/LZ4），上限 15–20 FPS；



可选：MJPEG（简化）、后续再评估 H.264（复杂度更高，放 vNext）。



消息主题：



viz.frame（帧流，二进制）/ viz.event（键鼠事件上行）/ viz.ctrl（启动/暂停/重置）/ viz.meta（分辨率、FPS、统计）。



背压：服务端对每连接 in-flight 窗口（如 32 帧）与全局令牌桶双限流；拥塞时优先丢弃最旧未处理帧。



录制与回放：关键帧 + 差分索引写入 MinIO，元数据进 Mongo；编辑器“时光机”与“观战/复盘”共用一份数据。



观战只读：弱网/家长观战走 SSE（只读帧/状态），避免双向信令带宽浪费。



4️⃣ 数据存储（冷热分层+可回溯）



MySQL（用户/订单/权限）+ MongoDB（课程/协作快照/可视化回放）+ MinIO（素材/导出工件/帧录像）+ Redis（会话目录/Presence/缓存）



MySQL：InnoDB / READ COMMITTED；启用 Binlog；金币/支付链路采用幂等键 + Outbox（本地事务出箱 → MQ）。



MongoDB：课程/作品采用“固定字段 + 扩展字段”；协作快照存增量 + 关键版本全量；回放索引与帧段指针入库。



MinIO：开启版本控制/生命周期/纠删码；跨区异地容灾。



Redis：presence:{room}:{uid} TTL 在线标记；房间成员集合与路由索引分离；禁止存储大消息体。



5️⃣ 网络与部署（Kong 对外统一）



Kong（Ingress/APIGW） + Kubernetes + Argo Rollouts + KEDA + Prometheus + Grafana + Loki + Tempo（OTel） + Sentry + CDN



WebSocket 关键配置



透传：Upgrade: websocket、Connection: upgrade



粘性会话：基于 sessionId/uid 的一致性哈希（自定义头），避免重连漂移；



限流与背压：按连接/主题令牌桶；设定 maxFrameSize；长流（AI/日志/帧）强制分片；



压缩与保活：按需 permessage-deflate；心跳 20–30s；超时 60–90s；



SSE 只读兜底：公告/观战/回放优先 SSE；写操作始终走 WS。



对外仅暴露 Kong；如需内部静态加速可用内网 Nginx，不对外暴露。



弹性与节点分池



KEDA：基于 RabbitMQ 积压、Celery 活跃任务、WS 会话指标自动扩缩。



节点池：ws-gateway 独立节点池（更高 ulimit/句柄/NIC），计算/存储分离。



6️⃣ AI 智能模块（单入口·策略路由）



文本/代码：通义千问 / 文心一言 / 星火（MVP 先接一家）；



语音：科大讯飞 TTS / 腾讯云 TTS（择一优先）。



AI Router：厂商路由、熔断、重试、配额/QPS、防雪崩；分片输出协议一致，编辑器无卡顿。



内容合规：厂商内容安全 + 自研策略（关键词/规则），违规降级或打回。



7️⃣ 安全与高可用（上线即位）



限流与防刷：IP/用户/API/WS 多维限速；黑白名单与动态风控。



灰度与回滚：Argo Rollouts（金丝雀 10%→30%→100%），探针覆盖建链率/RTT/关闭码 TopN/AI 成功率/帧延迟。



熔断与隔离：AI \& 渲染通道独立熔断阈值；Redis 保护；RabbitMQ 死信与限速。



数据安全：MinIO 周期快照；异地容灾；敏感字段加密。



API 安全：JWT + OAuth；请求签名（appKey + nonce + ts + HMAC），5 分钟重放窗。



交易保护：幂等键 + Outbox + 对账任务；失败仅前向补单。



一致性：Redis 状态持久策略 + MySQL 事务 + Mongo 回溯三位一体。



全链路追踪：OpenTelemetry → Tempo；前后端 Sentry 上报（采样可调）。



8️⃣ WS 协议与可视化帧（V1 规范）



通用信封：ver, sessionId, seq, topic, ts, payload, ackId?, traceId, win?, replaySince?, cmp?



按 topic 分片 seq（如 ai.\* / collab.\* / viz.\*），断线仅补该分片缺口；



ACK/重试：in-flight 窗口（建议 64）；指数退避，超限丢弃并记录原因码；



关闭码治理：4000–4999 业务码区间统一字典并上报。



可视化专用



viz.frame：frameId, ts, w, h, keyframe?, refId?, fmt=png|diff, dur, idx + 二进制数据；



viz.event：type=mouse|keyboard, payload...（上行）；



viz.meta：fpsTarget, fpsActual, dropped, queueDepth；



SSE 回放：text/event-stream，关键/差分帧的 URL 指针和时间戳索引。



9️⃣ SLO / 性能目标（MVP 验收扩展·含可视化）



长连稳定：30 分钟会话 p95 0 次非主动断开；



重连恢复：滚动重启 30% 网关实例时，90% 会话 ≤3s 恢复并补齐缺口；



广播延迟：集群公告 p95 ≤ 1s；



并发规模：支持 10 万 WS 长连（6–10 个网关副本，≥30% 余量）；



可视化 SLO（新增）：



帧端到端延迟 p95 ≤ 250ms（本地/近区），



编辑器内可视化 FPS p50 ≥ 15、丢帧率 p95 ≤ 5%，



回放启动时间 p95 ≤ 1.5s；



回滚成功率：金丝雀异常自动回滚 ≥ 95%；



观测面板必备：会话数、RTT p50/p95、重连率、关闭码 TopN、补发命中率、限流触发率、帧延迟/丢帧/队列深度、AI 成功率、支付回调成功率。



🔟 PK/荣耀协作（对战与观战的最小可靠面）



匹配服务：基于段位/胜率的 Elo 变体；



裁判服务：所有评测在服务端沙箱运行并回传判定；反外挂机制=“仅以服务器运行结果为准”。



对战通道：pk.state / pk.event / pk.result（WS）；观战走 SSE；



复盘：保留关键帧与输入事件序列（MinIO + Mongo 索引）；家长端只读观看、点赞/打赏通过 HTTP 侧路。

六、代码对战 · PK功能（新增核心玩法）



功能描述：

两个学生可以发起实时对战，系统自动生成一组编程任务（支持选择难度），比拼谁更快、更正确地完成。



玩法机制：

1\. 发起PK → 可邀请好友，也可匹配同等级玩家。

2\. 任务发布 → 双方同时开始，倒计时开始。

3\. AI实时检测 → 谁先正确完成，谁获胜。

4\. 获胜奖励 → 获胜方获得金币，失败方获得参与奖励。

5\. 对战排行榜 → 展示历史胜率、对战次数、连胜记录。



特色：

\- 反外挂机制，AI实时校验代码。

\- 家长端可观看PK过程，支持点赞和打赏。

\- 对战结束自动生成复盘，展示正确解法与优化建议。

\- 支持快速复赛、好友约战、匹配模式。



七、英语辅助功能（核心功能）



\- 深度集成编辑器与文档。

\- 支持单词悬停翻译、发音、报错翻译。

\- 家长端同步查看孩子的英语掌握情况。
